# references/issue-rules.md — 问题管理：立案、分级、修复判据、状态机

## 1. 发现即立案（file-first，禁止静默修复）

任何问题——**包括修复途中顺手发现的**——必须先写入 `ISSUES.md` 立案，再决定处理方式。否则总账对不上、无法审计。条目模板见 `templates/ISSUES.md`，逐条追加、勿用表格（便于原地改状态）。

> **禁止「契约未禁止」自行结案（HARD）**：程序**静默接受异常输入**（负数 / 零 / 越界 / 空值 / 特殊值）时，即使行为看似可解释为某种合理语义（如"负数=退款"），也**必须立案**——语义存在两种以上合理解释的，径直标 `NEEDS_CONFIRMATION` 走 MoA。把这类观察埋进轮次日志"记观察不立案"，等同于替用户拍板产品语义，属分流违规。

> **期望来源阶梯（oracle ladder）**：判定「期望行为」按优先级推导：规范/需求文档 > 代码注释与类型 > 已有测试 > 同类项目稳定惯例 > 通用合理性。逐级都推不出 = 该行为**无 oracle**：立案标 `NEEDS_CONFIRMATION`，不得自造期望后按己意判罚。

## 2. 分级 P0-P3（全项目统一 P 标度）

| 级别 | 判据 | 参考文档等价 |
|------|------|------|
| **P0** | 崩溃 / 数据丢失或损坏 / 可利用安全问题 / 不可逆副作用 / 核心流程完全不可用 / 基线跑不通 | S0 |
| **P1** | 主要功能错误且无合理绕行 / 关键权限错误 / 公开文档的主示例跑不通（SDK·库·CLI 的入门路径即主要功能） | S1 |
| **P2** | 次要功能缺陷或明显 UX 问题，但存在可接受的临时绕行 | S2 |
| **P3** | 文案 / 轻微视觉·交互瑕疵 / 日志噪音 / 低概率且低影响 | S3 |

分级按**用户影响 / 可达性 / 发生概率 / 可恢复性**判定，**禁止为尽快停止而降级**。

问题类型字段：`BUG / UX / 数据一致性 / 安全 / 流程 / 隐蔽逻辑 / TEST_GAP / DOC / ENV`。

## 3. 复现与证据

- 复现步骤**必须写成可机械重放的命令或操作序列**——之后复验就是原样重放它。
- 保存最小且充分证据：精确步骤、测试数据、期望、实际、日志/错误、截图/录屏位置（能力允许时）、发生频率、受影响范围。
- 先查重；同一根因的多个症状可关联，但**不要把互不相关的问题强行合并**。
- 发现 P0：立即保存证据并停止会扩大损害的同类操作，先隔离测试数据与受影响路径；仅在完全本地、可逆、不扩大损害时做最小复现与修复验证，同时继续其他安全独立的测试。

## 4. 立即修复判据（FR-4.5：全部满足才修）

1. 已**稳定复现**；
2. **期望行为明确**；
3. **根因定位有证据**；
4. 改动**局部、可逆、在任务范围内**；
5. **不改变**产品策略 / 公开契约 / 安全边界；
6. **能够验证**。

> **FR-4.6**：满足直修判据时，若已存在推荐修复方案（issue 追踪、文档、上游 patch），**优先采用推荐方案，不另起炉灶**。

以下情况**不擅自决定**，逐条记为 `NEEDS_CONFIRMATION`（继续其他测试）：产品需求/UX 取舍有多种合理答案；需要大规模重构、重设计、破坏性 API/DB 迁移或主要依赖升级；涉及认证授权/隐私合规/计费/数据保留/兼容承诺；需要真实凭据、外部写权限、生产数据或超沙箱资源；无法稳定复现，或修复风险明显高于问题本身。凡涉及**产品决策**一律进此类，绝不擅自替用户拍板 → 走 MoA（`references/moa-decision.md`）。

## 5. 修复节奏与回归保护

**修一个 → 验一个 → 提一个**。修根因不遮症状，改动最小但完整，遵循现有架构与风格。修复前或同时添加**能在旧代码上失败、修复后通过**的回归断言；无法自动化则记录原因并给可重复手工验证步骤。不删/不放宽/不跳过原有测试。详见 `references/loop-round.md` 第 4 步。

## 6. 同根因尝试上限（FR-1.5）

同一根因最多做 **3 次有实质差异**的修复尝试。仍失败 → 记录尝试、证据与下一步，标 `BLOCKED` 或 `NEEDS_CONFIRMATION`，转去测试其他区域。**禁止无变化地反复试错。** 转移前**先回滚工作树中失败尝试的残留改动**——「修一验一提一」保证已提交部分干净，但未提交的半成品 diff 同样不得跨问题遗留，残留会污染后续轮次的复现与证据。

## 7. 状态机

```
OPEN → FIXING → FIXED_UNVERIFIED → VERIFIED
                       ↓
   NEEDS_CONFIRMATION / BLOCKED / WONT_FIX / CANNOT_REPRODUCE
```

- `VERIFIED` **只能由原样重放复现步骤产生**（重放命令与输出同轮记录于 `runs/round-N.md`）；未重放不得标 VERIFIED。
- `NEEDS_CONFIRMATION` / `BLOCKED` / `WONT_FIX` / `CANNOT_REPRODUCE` **必须有完整证据与明确下一步**，不能当作方便停止的「问题停车场」。
- `CANNOT_REPRODUCE` 必须附尝试记录，不许编故事。

## 8. 疑似密钥 / 敏感数据（FR-6.6）

不读取、打印、复制、提交密钥与敏感数据。发现疑似密钥**只记录文件位置与风险类型，值一律脱敏**（如 `sk-****`）。绝不把密钥写入 `ISSUES.md`、日志或 commit。

## 9. 建议类 → SUGGESTIONS.md

新方向 / 新功能建议不进 `ISSUES.md`，存 `SUGGESTIONS.md`，每条附一条 MoA 决策建议链接（`decisions/DEC-NNN.md`）。**不擅自实现建议类内容**。
