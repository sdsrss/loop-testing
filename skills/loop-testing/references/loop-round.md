# references/loop-round.md — 每一轮的强制五步闭环

> 每轮开始前先重读 `STATE.md` / `ISSUES.md` / `FEATURE_MATRIX.md`——**一切进度以文件为准**。
> 五步：选场景 → 像真实用户使用 → 发现即立案/复现/分级 → 分诊修复+回归 → 复验+轮末结算。

## 第 1 步 · 选择本轮场景

优先顺序：**未覆盖功能 → 核心/高风险旅程 → 最近修改的功能 → 上轮修复的相邻区域 → 跨功能组合 → 易被忽略的边界与恢复路径**。

**实质差异要求**：每轮必须包含真实操作，且与上一轮在场景 / 数据 / 角色 / 操作顺序 / 视口 / 失败条件中**至少一项实质不同**。禁止连续两轮机械重复同一组脚本来凑停止条件。任何「候选退出轮」必须做**全功能回归**。

## 第 2 步 · 像真实用户一样实际使用

不要只读代码、只跑已有测试、只看页面能否打开。**必须尽量从真实用户可见入口实际使用产品**，代码/日志/自动化检查只作辅助定位（黑盒发现优先，白盒定位）。

按形态选驱动方式：

- **CLI**：从安装/启动与 `--help` 开始，实际跑典型工作流；验证参数、stdin、交互、退出码、stdout/stderr、文件副作用、错误输入、重复执行、中断与恢复、管道与重定向、TTY 与非 TTY（重定向/管道下颜色·进度·交互提示应降级而非挂死）、信号（中断/终止后的退出码与状态残留）。
- **API/服务**：启动后用 `curl` 等**真实调用**公开契约；验证认证与角色、正常 CRUD/业务流程、校验错误、状态码/错误体、分页、幂等或并发、持久化。**只用本地/模拟端点**。
- **Web/UI**：环境有可用无头浏览器（如 playwright）→ 真实点击、输入、键盘导航、前进后退、刷新、取消、重试；查桌面与窄屏、加载/空/错误/成功态、焦点、提示、控制台错误与失败请求。**无无头浏览器 → 退化为「API 调用 + 构建 + 页面渲染检查」，并把无法真实点按的部分如实记为「测试盲区」写进报告，不许假装测过**。
- **SDK/库**：以消费者身份按公开文档安装/导入/调用，覆盖公开 API、典型组合、错误处理、兼容范围。
- **插件/扩展**：在真实或可模拟的宿主中走完整生命周期——安装/加载、宿主调用、配置变更、禁用/卸载清理、升级迁移；验证宿主 API 契约与沙箱限制内的行为。无法起真实宿主 → 用宿主行为模拟脚本实跑，未覆盖的宿主面如实记为测试盲区。
- **数据/批处理**：用规模递进的真实输入文件/流实跑；验证幂等（同输入重复跑）、中断续跑、大文件、编码与损坏输入的失败行为、输出可重现。
- **多入口产品**：至少完成入口之间一条端到端旅程，验证状态一致性。

> **长驻服务 · 进程与 `.pids` 纪律（Web/API/后台任务）**：启动本地服务后，PID 必须**校验后**再记入 `docs/looptesting/.pids`——`$!` 可能与真正监听端口的进程不一致（如旧进程仍占端口），用 `lsof -t -i :PORT`（或 `ss -ltnp`）交叉确认「存活且在监听目标端口」，不一致时以端口定位的 PID 为准记录。每轮末 / 切场景重启服务前，按 `.pids` **与**端口双路径清理旧进程；遇 `EADDRINUSE` 用端口定位残留（`kill $(lsof -t -i :PORT)`）并在 `runs/round-N.md` 记一条环境事故（现象 / 定位 / 处置），避免旧进程静默服务请求污染证据。

**误用清单（每轮至少抽样若干）**：空输入 / 超长输入 / 特殊字符·emoji·多语言 / 非法类型 / 越界或不存在的 id / 重复提交 / 乱序操作 / 中途取消或刷新 / 破坏前后步骤依赖 /（可行时）并发与超时 / 管道与重定向等环境边界 / 失败注入（依赖不可用·超时·磁盘满·权限不足——仅当可低成本安全模拟时执行，否则如实记为测试盲区，不许假装测过）。

**同时盯四类问题（四个视角）**：
1. **功能错误**：行为违反明确规范/承诺/稳定惯例。
2. **数据与状态一致性**：增删改后再查，验证持久化与一致性。
3. **用户体验**：报错是否人话、流程是否绕、有无明显卡顿、文案错漏、可访问性。
4. **安全隐患**：注入、越权等——**只做记录与最小化本地验证，不做真实攻击/扫描/洪泛**。

## 第 3 步 · 发现即立案 / 复现 / 分级

见 `references/issue-rules.md`。要点：任何问题（含修复途中顺手发现的）**先写 `ISSUES.md` 立案再处理**，禁止静默修复；先复现、存最小充分证据；同一根因可关联，互不相关的不强行合并；按 P0-P3 分级，**禁止为收敛降级**。发现 P0 立即停止会扩大损害的同类操作，先隔离受影响路径。

## 第 4 步 · 分诊修复 + 回归保护

直修判据与状态机见 `references/issue-rules.md`。修复节奏与红线：

- **修一个 → 验一个 → 提一个**，再动下一个。严禁一次改多处再统一提交。
- 单个修复流程：先加**能在旧代码上失败**的最小回归断言 → 修根因（非遮症状）→ 断言通过 → 全量测试与构建通过 → 单独 commit：`fix(qa): [ISSUE-xxx] 简述`。仅当多问题同根因/同处代码可合并一个 commit，message 必须列全所有 ID。
- **commit 只许落在 qa 沙箱分支，且只含产品代码与测试**：`docs/looptesting/` 产物（STATE/ISSUES/runs 等）**绝不提交**，更绝不许任何提交落到用户主分支——提交前核对 `git branch --show-current` 与 staged 清单。
- **证据树 ≠ 提交树（worktree 模式，勿混）**：`docs/looptesting/` 证据产物始终留在**主工作树**（`sandbox-setup.sh` 有意如此——退出清 worktree 时证据才不丢）；而产品代码的编辑、运行与 `fix(qa)` commit 都发生在**沙箱 worktree**（qa 分支）里。两者是不同物理目录：别把台账写进 worktree，也别把代码 commit 落到主树。
- **不得**删除/放宽/跳过原有测试来迁就修复；**不得**顺手重构或格式化无关代码。
- 检查 diff：无敏感信息、无调试残留、无无关文件、无意外行为变化。若引入回归优先修正；无法修正则只撤销**你本轮自己的**改动（不碰用户内容），并把问题恢复为未解决。
- 用户可见行为/配置/操作方式改变 → 同步更新文档/示例/变更说明。
- 修复直接与可能间接影响的功能标 `RETEST_REQUIRED`；旧证据不自动算最新通过证据。

## 第 5 步 · 复验 + 轮末结算

- **复验 = 按台账记录的原复现步骤原样重放**，通过才可标 `VERIFIED`；未真正重放的一律不得标 VERIFIED（包括「看代码觉得肯定好了」）。不通过退回 `OPEN` 并记录。
- 先跑最小相关检查，再从**原始真实用户入口**重现同一路径，最后跑受影响范围内更广的回归。只跑单元测试不能把用户可见问题标 VERIFIED。
- 快速回归相邻功能，确认没修出新问题；修出的新问题回第 2 步立案。
- 更新 `FEATURE_MATRIX.md` / `ISSUES.md` / `runs/round-N.md` / `STATE.md`；重新检查健康状态与自动化检查，比对基线确认无新增失败。
- **STATE.md 更新粒度（防硬截断丢进度）**：不要只在轮末更新——**每次立案、每个修复提交、每次复验之后**，立即刷新 `STATE.md` 的「最后动作 / 下一动作」。会话可能因轮次上限或上下文截断在轮中硬停，只在轮末写 STATE 会让续跑者重复半轮工作。机器判读字段 `round` 仍只在整轮完成时递增。
- 输出一句精简进度：本轮覆盖（用例数，对比上轮）、发现（按级别 + ID）、修复与复验数、遗留、检查结果、下轮重点。**不要把中间进度伪装成最终结论**。
- 计算本轮是否为「收敛低风险轮」并更新 `converged_streak`（判据见 `references/exit-and-report.md`），然后决定进入下一轮还是停止。
